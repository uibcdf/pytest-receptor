# Prior art: other plugins with the same thesis

**2026-07-19.** Written after discovering, four days into publishing our own
claims, that two plugins already do what we do. Neither was known to us while
the 0.6 scope was written.

## Method

pytest's official plugin list is generated by scanning the PyPI simple index for
project names beginning with `pytest-` or `pytest_`
(`scripts/update-plugin-list.py`). All 2,016 entries were parsed and searched for
token-efficiency and agent-oriented summaries.

Everything below the survey table is **measured, not read**. Both plugins were
installed into isolated virtualenvs against pytest 9.1.1 and run on the same
scenarios as our own. Reading their source suggested the conclusions; running
them established the conclusions.

## The field

| Plugin | Version | Released | Summary |
| :--- | :--- | :--- | :--- |
| `pytest-agent-digest` | 0.3.2 | 2026-05-02 | "≥20% smaller in token count than raw pytest output" |
| `pytest-markdown-report` | 0.2.0 | 2026-07-15 | "Token-efficient markdown test reports for LLM-based TDD agents" |

Adjacent but not the same thesis:

| Plugin | What it actually is |
| :--- | :--- |
| `pytest-beacon` | CTRF reports written to file — a machine-readable interchange format, not console compaction |
| `pytest-kedge` | A DSL for *authoring* tests, with an agent summary attached |

Everything else matching "LLM" or "agent" in the list tests LLM applications.
That is the opposite direction: they put pytest around an LLM, we put an LLM
around pytest.

## Verified behaviour

pytest 9.1.1. `s5` = a file with no tests (exit 5). `s2` = a module with a
failing import (exit 2).

| | `pytest-markdown-report` | `pytest-agent-digest` | `pytest-receptor` |
| :--- | :--- | :--- | :--- |
| exit 5, nothing collected | `**Summary:** 0/0 passed` | one blank line | `NO_TESTS exit=5` |
| exit 2, collection error | `# Collection Errors` ✔ | *no output at all* | `COLLECTION_ERROR exit=2` |
| Derives verdict from `ExitCode` | never receives it | receives, never uses | yes |
| Sets `tbstyle` at configure time | no ✔ | no, respects the user ✔ | no ✔ |
| Uses `no_summary` | no ✔ | no ✔ | no ✔ |
| xdist handling | none | none | yes |
| Groups by root cause | **no** | **no** | **yes** |

### The founding bug, independently reproduced

`pytest-markdown-report` prints `0/0 passed` while pytest exits 5. A consumer
reading "passed" concludes the suite is green; what happened is that nothing
ran. This is the defect this project was created to fix, in a plugin published
four days before this document.

Its `pytest_sessionfinish(self, session)` does not accept `exitstatus`, and the
summary is built from counted reports:
`f"{total_passed}/{total} passed"` (`src/pytest_markdown_report/plugin.py:393`).

`pytest-agent-digest` fails differently and, on the collection error, worse: it
emits nothing whatsoever, so the consumer is told neither the outcome nor the
traceback. Its `pytest_sessionfinish` **does** take `exitstatus`, and documents
it in the docstring, but `render_report()` is called without it
(`pytest_agent_digest/plugin.py:55`).

### Where they are not wrong

Both avoid the two traps that cost us the most: neither sets `tbstyle` at
configure time, and neither uses `no_summary`. That is architecture, not luck —
we hit those because we wanted to silence pytest while keeping its data. One of
them replaces the writer, the other only appends. Different design, different
hazards.

`pytest-markdown-report` handles the collection error cleanly, which we should
not pretend otherwise.

## Honest position

**Where we win.** Root-cause grouping, which neither has, and a verdict derived
from `pytest.ExitCode`, which neither does.

**Where we do not.** On a single isolated failure we measure −16.2% against a
tuned pytest. That is inside the range they claim. We are not better at
compressing an individual failure; we are better at not repeating one bug
thirty-eight times. Our headline figures come from cascades, and a suite whose
failures do not cluster will not see them.

Stating this precisely matters more than it costs. A vague claim of superiority
is undone by the first benchmark someone else runs.

**Where we are behind.** `pytest-markdown-report` has `--markdown-rerun-cmd`, so
a project driven by `just test` or `uv run pytest` gets a rerun command that
works. We always emit `pytest ...`, which does not paste-and-run everywhere —
and "the command works pasted verbatim" is our most repeated promise. Their idea
is better than ours. Filed as a proposal; the implementation is ours to write.

## Decisions

**No comparative benchmark will be published.** Our scenarios favour our
architecture by construction, so publishing head-to-head numbers would be
marketing dressed as measurement. The figures above stay internal. Public
documentation may acknowledge that these plugins exist; it may not rank them.

**Their defects were not reported to them, yet.** Deferred 2026-07-19 by
decision, not by oversight. The evidence is preserved here so the reports can be
filed without re-deriving them. Reporting them is the correct move and is
unrelated to competition — we filed two findings against pytest itself this
week on the same principle.

**No novelty claims to retract.** The README and documentation argue from the
problem and from measured figures, never from being first or only. Checked.

## Reproducer

```bash
mkdir s5 s2
echo '# no tests here'                                  > s5/test_none.py
printf 'import nonexistent_xyz\ndef test_a(): assert 1\n' > s2/test_broken.py

python -m pytest s5   # markdown-report: "0/0 passed", pytest exits 5
python -m pytest s2   # agent-digest --agent-digest=term: no output, pytest exits 2
```

Capture the exit code from pytest itself, not from the tail of a pipe — the
first attempt at this measurement reported the status of `tail` and had to be
redone.

## Consequences

That two independent projects derive the verdict from collected reports rather
than from the exit code strengthens the case in
[pytest#14710](https://github.com/pytest-dev/pytest/issues/14710): it is a trap
the extension surface invites, not a mistake three people happened to make. If
that argument is ever made publicly, it should be made after telling them, not
by citing them as casualties.
